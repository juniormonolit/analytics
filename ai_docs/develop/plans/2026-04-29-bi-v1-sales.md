## Goal

Build the v1 BI analytics application: Russian-language Next.js (App Router) + TypeScript + Tailwind + Supabase app, scoped to the **Sales** section with two reports — **By Managers** (`by-managers`) and **By Product Groups** (`by-product-groups`). Implement the fixed runtime described in `ai_docs/01_PRODUCT_SPEC.md` (filters → reports/run API → table with comparison & totals → drill down), with localStorage-backed user preferences ready to be swapped for a Supabase table later.

## Non-Goals (Out of Scope for v1)

- Other sections (Найм, Маркетинг, Реализация) — render disabled in sidebar only.
- Universal dataset / chart builder.
- Complex permissions, AI layer, public dashboards.
- Drag-and-drop dashboard grid, Excel auto-import.

## Architecture Decisions

### Folder Layout (top-level, no `src/`)

```
app/                          # Next.js App Router routes & API
  api/
    catalog/teams/            # tree-or-flat teams catalog
    catalog/metrics/          # metrics catalog
    reports/run/              # POST /api/reports/run
    reports/drilldown/        # POST /api/reports/drilldown
    health/db/                # smoke check
  sales/
    layout.tsx                # mounts FilterBar + ReportTabs
    by-managers/page.tsx
    by-product-groups/page.tsx

components/                   # presentational + composite UI
  shell/                      # AppShell, Sidebar, ReportTabs, SectionHeader
  filters/                    # FilterBar, DateRangePicker, DepartmentTreeFilter
  reports/                    # ReportTable, ReportToolbar, MetricPickerModal,
                              # DrillDownPanel, states/
  ui/                         # ThemeToggle and other primitives

features/                     # sales-specific business logic
  sales/state/                # Zustand stores (filters, reportPrefs, drilldown)
  sales/hooks/
  reports/engine/             # runReport, queryDeals, aggregate, comparison,
                              # totals, grouping, dimensions, schema, types
  reports/drilldown/          # level handlers + types + schema

lib/                          # framework + cross-cutting helpers
  supabase/                   # client.ts, server.ts, types.generated.ts
  env.ts                      # zod-validated env
  period/                     # defaults, presets, format
  format/                     # money, percent, number formatters (ru-RU)
  theme/                      # tokens + ThemeProvider
  navigation/sections.ts      # sidebar config

repositories/
  reportPreferences/          # interface + LocalStorage impl (swap-ready)
```

### Key principles enforced

- **Russian-language UI** everywhere (except common technical terms).
- **No hard-coded colors** — every color comes from a CSS variable defined in `app/globals.css` and surfaced through Tailwind theme tokens (`bg-bg-primary`, `border-border-primary`, `text-text-primary`, `accent-primary`, `table-header-bg`, `scrollbar-thumb`, etc.).
- **Server-only Supabase reads for sensitive aggregates** — `/api/reports/run` and `/api/reports/drilldown` use `lib/supabase/server.ts`. Catalogs (teams, metrics) may be read from the server too for consistency.
- **Fixed runtime, not a BI builder** — no generic dataset/dimension abstractions beyond what the two reports need.
- **Repository pattern for preferences** — interface + LocalStorage impl now, Supabase impl later (no call-site changes).
- **Period math is pure** — `lib/period/*` is fully unit-testable and free of UI/Supabase dependencies.

### State Management

- **URL search params** — canonical source of truth for `period`, `comparisonPeriod`, `teamIds` (so links are shareable).
- **Zustand store** — section-scoped mirror that hydrates from URL on load and pushes back on change (debounced).
- **TanStack Query** — caches `/api/reports/run`, `/api/reports/drilldown`, and catalog endpoints.
- **localStorage (via repository)** — only user preferences (metric selection, ordering, hidden columns, widths, grouping, sort).

### Data Contracts (locked by the docs)

- Period filter on `sa.deals.created_at` with inclusive `to` (`< to + 1 day`).
- Department filter pushes the full expanded leaf set (`team_id = any(...)`).
- Comparison runs the same query twice and merges by dimension key.
- Ratio/CR metrics are recomputed from numerator/denominator after aggregation; never averaged.

### BI-004 — FilterBar: DateRangePicker + DepartmentTreeFilter + section state

Section-level FilterBar mounted in the sales layout. Holds period, comparisonPeriod, teamIds. Defaults: current = 1st of current month..yesterday, comparison = same-length tail of previous month (e.g. today=29.04.2026 ⇒ current 01.04..28.04, comparison 03.03..31.03). Presets: Сегодня, Вчера, Эта неделя, Прошлая неделя, Этот месяц, Прошлый месяц.

`DepartmentTreeFilter` loads teams from a server route `/api/catalog/teams` that **at runtime** detects whether `sa.teams` has a parent column (`parent_id` or `parent_team_id`); if yes, build the tree, otherwise return a flat list. Tree supports expand/collapse, partial-state checkboxes, parent-selects-children, and emits `selectedTeamIdsExpanded` (resolved leaf list).

State lives in a Zustand store (`features/sales/state/filtersStore.ts`) with URL search-param mirroring (`from`, `to`, `cfrom`, `cto`, `teams`); switching tabs preserves filters.

**Acceptance criteria:**

- Defaults match the spec (unit-tested in `lib/period/__tests__/defaults.test.ts`).
- All six presets correctly update both `period` and `comparisonPeriod`.
- DepartmentTreeFilter button shows `Отделы (N)`; partial-state parents render indeterminate.
- Server route returns `{ kind: 'tree', nodes }` or `{ kind: 'flat', teams }` based on schema introspection.
- URL ↔ store sync survives reload; switching `/sales/by-managers` ↔ `/sales/by-product-groups` keeps filters.
- All labels in Russian; no hard-coded colors.

### BI-005 — Report engine API: `/api/reports/run`

Implement `POST /api/reports/run` per `ai_docs/03_REPORT_ENGINE.md`. Reads `sa.deals` filtered by `created_at` (inclusive `to`) and optional `team_id = any(:teamIds)`. Loads metrics from `sa.metrics` (in-memory cache per process), respecting `aggregation_fn` (sum/avg/none) and recomputing ratio/CR metrics from their numerator/denominator (`dependencies`). Two report slugs:

- `by-managers` — dimension `manager_id` + `manager_name` (join `sa.employees` on `bitrix_id = current_manager_id`, fallback to id), `team_id`, `team_name`.
- `by-product-groups` — dimension `product_group_id` + `product_group_name` (join `sa.product_groups`).

Runs the same query twice (period + comparisonPeriod), merges by dimension key, computes `delta` and `deltaPercent` (`null` if previous=0). Computes the totals row with proper CR recomputation. Supports `grouping ∈ { 'none', 'team', 'total' }`. Validates input with zod.

**Acceptance criteria:**

- Period filter `created_at >= from::date and created_at < to::date + interval '1 day'`.
- Both periods use identical filter logic; merge by dimension key.
- Calculated/CR metrics recomputed from numerator/denominator (never averaged).
- `totals` row recomputes CR; ratio with denominator=0 returns `null`.
- Grouping `team` aggregates by team using the same correct CR logic.
- Response shape exactly matches the contract in `03_REPORT_ENGINE.md`.
- Zod-validated request body; invalid bodies → 400 with structured error.
- Server uses `lib/supabase/server.ts` only.
- Unit tests cover merge, delta edge cases, totals CR, team grouping.

**Verification:** `curl` with the sample body returns `{ columns, rows, totals, meta }`; Vitest passes.

## Implementation Notes

- Prefer **server route handlers** for all data access — even catalogs — so we keep one consistent path and avoid bundling Supabase fetch logic into client components beyond what TanStack Query needs.
- Keep the engine pure: `aggregate.ts`, `comparison.ts`, `totals.ts`, `grouping.ts` should be testable in isolation with plain row arrays as inputs.
- Use `Intl.NumberFormat('ru-RU', ...)` and `Intl.DateTimeFormat('ru-RU', ...)` everywhere for formatting.
- The `DepartmentTreeFilter` should not assume hierarchy exists — its API contract returns `kind: 'flat' | 'tree'`, the component renders both modes.
- All disabled sidebar items must still be **announced** as disabled (aria-disabled, focus-skipped) for accessibility.
- Theme toggle uses an inline `<script>` in `<head>` reading `localStorage.getItem('bi.theme')` and setting `data-theme` before paint to avoid FOUC.
